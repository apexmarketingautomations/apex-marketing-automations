## CORE TRUTH
Do NOT confuse:
- working backend
with
- finished product

Replit and most builders often stop at:
- endpoints built
- APIs connected
- data stored
- pipeline works

and then act like the feature is done.

That is NOT my standard.

If the engine works but there is:
- no steering wheel
- no dashboard
- no visible controls
- no proper UI
- no clear user flow
- no easy way for a human to use it

then the feature is **not complete**.

### My completion rule:
A feature is only done when:
- backend works
- frontend/UI exists
- the feature is visible
- the feature is usable by a real person
- it is connected end-to-end
- it fits naturally into the product
- it makes sense in the UX
- it has been logically tested
- it feels launch-ready

If a user cannot see it, click it, manage it, understand it, or access it easily, it is not done.

## REQUIRED BUILD FRAMEWORK
Every feature must be evaluated across all 5 layers:

### 1) Backend
- routes
- business logic
- db/schema
- auth/permissions
- integrations
- storage
- automations
- webhooks

### 2) Frontend
- pages
- sections
- tabs
- cards
- buttons
- forms
- filters
- modals / drawers
- nav visibility
- settings surfaces
- empty states
- loading states
- error states

### 3) Connection
- frontend actually talks to backend correctly
- real data is shown
- actions work correctly
- create/edit/delete/update flows work
- statuses display correctly
- no fake placeholder behavior

### 4) UX
- easy to find
- easy to understand
- easy to use
- clear primary action
- no dead ends
- no hidden features with no usable surface
- strong information hierarchy

### 5) Testing
- verify the real flow logically
- do not assume
- confirm what works
- identify what still needs proof
- separate “verified” from “should work”

If ANY of these layers are missing, the feature is **not complete**.
