Build a production-grade tracking and intelligence service for Apex Marketing Automations in Replit.

This system must not be a disconnected tracker. It must function as an event collection, attribution, and learning layer that feeds the Apex Intelligence Database so Apex can improve over time across clients, cards, campaigns, funnels, and lead behavior.

PRIMARY OBJECTIVE:
Track NFC business card taps, QR scans, page views, CTA clicks, form starts, lead submissions, bookings, and downstream outcomes, then normalize and send that data into the Apex Intelligence Database so the platform can learn patterns, score intent, improve routing, and power future automation decisions.

TECH STACK:
- Node.js + TypeScript
- Express
- PostgreSQL
- Prisma ORM
- Zod validation
- Redis if needed for caching, rate limits, replay protection, and queues
- Background job support for async processing
- Secure environment-based configuration
- Clean modular architecture

CORE SYSTEM PRINCIPLES:
1. Fast event capture
2. Durable attribution across the full funnel
3. Clean normalization into Apex Intelligence Database
4. Extensible intelligence layer for learning and optimization
5. Production hardening and security
6. No broken attribution chains
7. Multi-tenant design for multiple Apex clients

ARCHITECTURAL LEVELS:

LEVEL 1: CAPTURE LAYER
Purpose:
Receive traffic from NFC cards, QR codes, digital business cards, landing pages, and forms without losing attribution.

Requirements:
- Create short tracking routes like /t/:slug
- Each slug maps to a specific card, campaign, client, and destination
- On each request:
  - validate slug
  - create visit record
  - generate visit_id
  - generate signed attribution token
  - store source metadata
  - redirect quickly to destination
- Support both NFC and QR as first-class entry methods
- Preserve attribution parameters across redirects and page navigation
- Capture events:
  - tap
  - qr_scan
  - page_view
  - cta_click
  - form_start
  - lead_submit
  - booked_call
  - qualified_lead
  - closed_sale if later sent from Apex CRM
- All public capture endpoints must be extremely fast and resilient

LEVEL 2: ATTRIBUTION LAYER
Purpose:
Connect every interaction to the correct client, campaign, card, lead, and conversion path.

Requirements:
- Maintain a durable attribution chain from first tap/scan to final lead outcome
- Every visit must link to:
  - client_id
  - campaign_id
  - card_id
  - destination_id
  - visit_id
- Every tracked event must link back to the original visit when possible
- Support identity stitching across:
  - repeated visits
  - same device
  - form submissions
  - CRM lead creation
  - booked calls
- Add signed attribution tokens so query params cannot be easily spoofed
- Support idempotency for conversion events
- Support server-to-server events from Apex forms, workflows, and CRM automation
- Allow manual reconciliation when attribution is incomplete

LEVEL 3: INTELLIGENCE LAYER
Purpose:
Feed structured, normalized event data into the Apex Intelligence Database so Apex can learn and improve over time.

This is critical:
The system must write normalized event, attribution, lead, and performance data into a dedicated Apex Intelligence data model that can later be used for:
- lead intent scoring
- campaign optimization
- best-performing card analysis
- best-performing CTA analysis
- industry pattern recognition
- time-to-conversion analysis
- client benchmarking
- routing optimization
- follow-up optimization
- sales pattern detection
- A/B learning
- future AI-driven recommendations inside Apex
