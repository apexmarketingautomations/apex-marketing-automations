# Production-Safe Integration of God Mode as Apex’s Canonical Control Plane

## Executive summary

A production-safe “God Mode” integration is not a UI refresh; it is a **control-plane refactor** that must preserve tenant isolation, eliminate duplicate sources of truth, and roll out via **backward-compatible (“parallel change”) steps** with **feature-flag kill switches** and **canary rollout** to minimize blast radius. Tenant isolation must be treated as foundational (“not optional”) at every layer—routing, API authorization, and data access—because a single cross-tenant bug is a platform-ending incident. citeturn0search2turn2search3turn3search0

Because no Apex production URL or internal API docs were provided, this report uses **generic, safe defaults** and references established primary sources for zero-downtime migrations (Parallel Change / expand–migrate–contract), feature-flag release control, canarying strategy, tenant isolation, and reliable eventing patterns. citeturn1search0turn0search1turn0search3turn1search13turn2search3

The core architectural decision that makes God Mode “canonical” without breaking existing flows is:

**God Mode provisions and governs → Sub-accounts operate and persist canonical operational records.**  
God Mode may *seed* initial values (e.g., website URL), but after provisioning the **sub-account owns the operational record** (website source, training state, widget state), while God Mode remains the authoritative **orchestration and standards** layer (blueprints, deployments, fleet health, incident response).

Rollout should follow a phased **expand → migrate → contract** approach (Parallel Change) to avoid downtime and allow rollback at most steps. citeturn1search0turn1search20turn3search6

## Data-model audit plan with canonical records, required fields, and zero-downtime migrations

### Canonical records required

To make God Mode canonical without duplicating operational truth, you need a small set of **explicit, canonical records**. This follows tenant-isolation discipline: everything is tenant-scoped (subAccountId/tenantId), and God Mode never stores “operational duplicates” it can’t reconcile safely. Tenant isolation must be enforced rigorously. citeturn0search2turn2search3turn3search0

**Canonical operational records (sub-account owned):**
- `Tenant` / `SubAccount`
- `WebsiteSource` (the canonical website URL + origin + operational status)
- `WebsiteSyncRun` (append-only runs for auditability)
- `AiTrainingRun` (append-only runs)
- `WidgetDeployment` (embed key/config + status)
- `ProvisioningRun` (tracks God Mode provisioning steps and outcomes)

### Migration strategy

Use a **Parallel Change / expand–migrate–contract** approach where new schema coexists with old until all code and data are migrated. This is explicitly the safe pattern for breaking interface changes and is commonly used for zero-downtime evolution. citeturn1search0turn1search20turn3search6

If your system is Postgres, be mindful that DDL can still lock; understand safe operations for your DB version. For example, Postgres documents that adding a column with a constant default is optimized (no row-by-row rewrite), while volatile defaults can force row updates. citeturn1search2

### SQL migration pseudocode (idempotent outline)

Assumptions:
- Existing table has `tenants` and possibly `tenant.website_url` or similar.
- You’re adding canonical `website_sources` without breaking existing code.
- You want **idempotent scripts** capable of re-running safely.

**Expand phase (safe additive changes)**

  tenant_id uuid NOT NULL,
  website_url text NOT NULL,
  website_domain text NOT NULL,
  origin text NOT NULL,
  ownership_scope text NOT NULL DEFAULT 'SUBACCOUNT',
  status text NOT NULL,

  training_status text NOT NULL,

  last_training_completed_at timestamptz NULL,

  updated_at timestamptz NOT NULL DEFAULT now()

CREATE INDEX IF NOT EXISTS idx_website_sources_tenant ON website_sources(tenant_id);
CREATE UNIQUE INDEX IF NOT EXISTS uq_website_sources_tenant_active

**Application-level backfill runner (recommended)**
- Iterate tenants in ascending order of `id` (or `created_at`).
- For each tenant with an existing website URL and no active website_source:
  - Insert website_source with `origin='GOD_MODE_ONBOARDING'` if that’s where it came from; otherwise infer a safe default (`LEGACY_IMPORTED`).
  - Set status to `CONNECTED_UNSYNCED` (explicitly requires sync action).
  - Do NOT start sync/training automatically unless you have capacity controls.

**Contract phase**
- After you have dual-read/dual-write stabilized and verified, remove legacy fields and code paths.

### NoSQL migration pseudocode (Mongo-like)

**Expand**
- New collection: `websiteSources`
- Unique compound index on `{ tenantId: 1, active: 1 }` to prevent duplicates.

**Backfill**
```js
// idempotent_backfill_website_sources.js
const cursor = db.tenants.find(
  { websiteUrl: { $exists: true, $ne: "" } },
  { _id: 1, websiteUrl: 1, createdAt: 1 }
).sort({ _id: 1 });

for (const t of cursor) {
  db.websiteSources.updateOne(
    { tenantId: t._id, active: true },
    {
      $setOnInsert: {
        _id: uuid(),
        tenantId: t._id,
        websiteUrl: t.websiteUrl,
        websiteDomain: extractDomain(t.websiteUrl),
        origin: "LEGACY_IMPORTED",
        ownershipScope: "SUBACCOUNT",
        status: "CONNECTED_UNSYNCED",
        syncStatus: "NOT_STARTED",
        trainingStatus: "NOT_STARTED",
        widgetStatus: "NOT_GENERATED",
        createdAt: new Date(),
      },
      $set: { updatedAt: new Date() }
    },
    { upsert: true }
  );
}
```

- A single `migrations` runner that:
  - records applied migrations in a dedicated table/collection (Flyway/Liquibase-style schema history is the standard pattern) citeturn2search4turn2search29
  - ensures each migration is additive and backward-compatible
  - supports re-runs (idempotent DDL where possible; application-level upserts for data)
- If you use Flyway/Liquibase, follow their immutable migration practices and re-runnable patterns for views/procedures as needed. citeturn2search0turn2search1turn2search25

## Deterministic handoff rules with state machines and exact enums

### Deterministic handoff rules

These rules eliminate drift by making the boundary explicit.

**Rule group: provisioning → operational ownership**
- God Mode may **seed** website URL during provisioning.
- Once the sub-account exists, the website URL becomes a **sub-account canonical `WebsiteSource`** record.
- God Mode is not allowed to maintain a second operational website record; it may only:
  - write via the same canonical API used by sub-accounts (with elevated permissions), or
  - create a provisioning request that is executed into the canonical records.

**Rule group: interface compatibility**
- During migration, you may dual-write to old and new stores temporarily (Parallel Change), but reads should be deterministic:
  - Prefer `WebsiteSource` if present; fallback to legacy field only if missing.
- Dual writing and/or backfill should be used to keep stores consistent until cutover, following proven online-migration practice. citeturn1search0turn3search6

**Rule group: event consistency**
- If you emit events (sync requested, training completed), avoid dual-write inconsistency by using the **Transactional Outbox pattern**: write business state + outbox event in one DB transaction, publish asynchronously. citeturn1search13turn1search9

## Backend implementation tasks with API contracts, event flows, rollback, feature flags, and migration windows

### Feature-flag gating strategy (non-negotiable)

Use feature flags as a release safety mechanism: enable new behavior for internal users first, then canary, then full rollout. Feature toggles are explicitly designed for safe, rapid delivery and rollback. citeturn0search1turn3search1turn0search3

Minimum flags:
- `websiteSourceV2_read` (new read path)
- `websiteSourceV2_write` (dual-write / new write path)
- `godModeControlPlaneNav` (show God Mode nav)
- `godModeProvisioningV2` (new provisioning wizard)
- `widgetGatedByTraining` (prevents widget generation on stale source)

  - Must have `status in {CONNECTED_SYNCED, NEEDS_RESYNC (if allowing), TRAINED}`
  - If stale, require `force=true` or block (your choice; recommended block).

### Event flows (reliable and tenant-safe)

Use an outbox to avoid the dual-write problem (DB update succeeds, event publish fails). The Transactional Outbox pattern is the standard approach: write business state + outbox record atomically, then publish. citeturn1search13turn1search9

Proposed events:
- `tenant.provisioning.requested`
- `tenant.provisioning.completed`
- `website_source.seeded`
- `website_sync.requested`
- `website_sync.completed`
- `ai_training.requested`
- `ai_training.completed`
- `widget.generated`

### Rollback strategy (zero downtime)

This plan assumes you’re using Parallel Change (expand → migrate → contract), which is explicitly designed so you can roll back code while keeping schema compatible. citeturn1search0turn1search20

Rollback levers, in order:
1) Disable feature flags (instant kill switches) to revert behavior without deploying. citeturn0search1turn3search1  
2) Stop background migrations/backfills (they must be idempotent and resumable).  
3) Keep expanded schema in place (don’t contract until stable).  
4) If needed, traffic-level rollback (blue/green or canary reversal) per your deployment system. Canarying reduces blast radius and supports controlled rollback. citeturn0search3turn0search15

### Migration windows (safe defaults)

- **Schema expand**: off-peak, but designed to be safe online.
- **Backfill**: throttled, chunked, runs continuously; pause/resume.
- **Cutover**: enable `websiteSourceV2_read` for internal users → canary tenants → full.
- **Contract**: only after an entire release cycle with zero fallback reads, plus explicit metrics.

  - create sub-account via God Mode provisioning
  - verify website_source seeded correctly
  - verify sub-account reads canonical record (no legacy drift)
- Outbox publishing correctness (event appears after DB commit).

**E2E tests (Playwright/Cypress)**
- God Mode: create sub-account with website URL → sub-account website page shows “already connected.”
- Sub-account: change URL → status becomes `NEEDS_RESYNC` → widget gated.
- Error paths: simulate crawl failure → UI shows recovery actions.

### Canary rollout steps

Canarying is explicitly recommended to reduce risk by initially exposing a small portion of production to a new release. citeturn0search3turn0search15

Suggested sequence:
1) Internal-only: enable flags for platform owner + test tenants.
2) Canary tenants: enable for 1–5% of tenants with low risk.
3) Expand: 25% → 50% → 100% while monitoring.
4) Contract: only after stable.

### Monitoring and alerts (minimum set)

Use Google SRE “monitoring distributed systems” principles and track at least the golden signals: latency, traffic, errors, saturation. citeturn1search3

Track metrics by tenant + endpoint:
- API error rate (4xx/5xx) for website-source endpoints
- latency p95/p99 for provisioning + sync requests
- sync/training job success rate
- queue depth / worker saturation
- cross-tenant authorization denials (should spike if probing occurs)
- “fallback read” rate (how often legacy website URL used)

Alerts:
- Any cross-tenant access anomaly
- sync failure rate > threshold
- training failure rate > threshold
- provisioning failure rate > threshold
- elevated p99 latency after enabling flags

## Risk matrix and mitigations

Tenant safety and no data drift are the primary constraints. AWS emphasizes isolation as foundational; OWASP enumerates multi-tenant risks like cross-tenant data leakage and impersonation. citeturn0search2turn2search3turn3search0

| Risk | Likelihood | Impact | Mitigation (actionable) |
|---|---:|---:|---|
| Cross-tenant data leakage (BOLA / tenantId tampering) | Medium | Critical | Server-side tenant scoping only; object-level authorization for every endpoint receiving IDs; deny-by-default; security tests. citeturn3search0turn2search3 |
| Duplicate website sources per tenant (drift) | Medium | High | Unique constraint/index on active website source; idempotent upserts; single canonical read path. |
| Legacy vs V2 drift during migration | Medium | High | Parallel Change with dual-write + backfill + metrics for fallback reads; contract only when fallback is zero. citeturn1search0turn3search6 |
| Outbox/event inconsistency (state updated, event missing) | Medium | High | Transactional outbox pattern; publisher retries; idempotency keys. citeturn1search13turn1search9 |
| Background job overload (crawler/training) after cutover | Medium | High | Rate limiting per tenant; queue controls; progressive enablement via canary rollout. citeturn0search3 |
| DDL locks / slow migrations (DB-specific) | Low–Medium | High | Use online-safe DDL; off-peak expand; chunk backfills; avoid volatile defaults; DB-specific playbooks. citeturn1search2 |
| UI confusion (God Mode vs sub-account ownership ambiguous) | Medium | Medium | Add Platform Control Architecture section once; make website origin visible; state-driven copy. |
| Rollback too slow / requires redeploy | Medium | High | Feature flags kill switch; keep expanded schema; avoid contract early. citeturn0search1turn3search1 |
| Audit trail gaps (can’t prove what happened) | Medium | Medium | Append-only `*_runs` tables; provisioning runs; event log correlation IDs. |
