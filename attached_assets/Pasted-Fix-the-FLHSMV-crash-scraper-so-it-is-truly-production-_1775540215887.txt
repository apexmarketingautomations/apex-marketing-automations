Fix the FLHSMV crash scraper so it is truly production-ready and demonstrably working end-to-end.

This is **not** a backend-only task. This is **not** complete unless ingestion is proven, records are stored, leads are created, logs are visible, and the flow is testable.

## CORE GOAL
The FLHSMV worker must reliably:
1. poll FLHSMV
2. fetch crash data successfully
3. parse it correctly
4. store new records in `crash_reports`
5. create leads from valid crashes
6. produce clear logs and proof that ingestion happened

## SUCCESS STANDARD
Do **not** mark this complete just because:
- the worker runs
- an endpoint responds
- the scraper “should” work
- a job completes with no visible output

This is only done if there is **real or simulated evidence** that the full ingestion flow works end-to-end.

## WHAT TO DO

### 1) AUDIT THE CURRENT INGESTION PIPELINE
Inspect the full FLHSMV flow first before changing anything:
- scheduler / worker trigger
- request construction
- auth / headers / params
- time window logic
- query filters
- parser / normalization logic
- dedupe logic
- insert into `crash_reports`
- lead creation logic
- failure logging
- retry logic

Identify exactly where the current failure is:
- request rejected
- empty results
- parser mismatch
- DB insert blocked
- dedupe too aggressive
- lead creation failing
- time window wrong
- filters too narrow
- silent failures / weak logs

Do not guess. Find the real break point.

---

### 2) FIX REQUEST PARAMS, FILTERS, AND TIME WINDOWS
Verify and correct:
- request params
- query payload format
- date/time ranges
- timezone handling
- rolling poll window logic
- source-specific filters
- pagination / batch logic if relevant

Make sure the worker does **not** miss incidents because:
- time windows are too tight
- timestamps are compared incorrectly
- timezone conversion is wrong
- filters exclude valid crashes
- dedupe suppresses fresh records

If needed:
- widen the poll window safely
- add overlap protection
- improve dedupe rules so overlap does not block real new incidents

---

### 3) HARDEN RELIABILITY
Add robust reliability logic:
- retry with exponential backoff for HTTP failures
- timeout handling
- non-200 response logging
- malformed payload logging
- parser failure logging
- DB insert failure logging
- lead creation failure logging
- job summary logging

Each worker run should clearly log:
- poll start time
- request sent
- params used
- response status
- count of items returned
- count parsed
- count inserted into `crash_reports`
- count skipped as duplicates
- count turned into leads
- count failed
- poll end time

No vague logs. No silent failures.

---

### 4) ADD TEST HARNESS / PROOF HARNESS
Build a test harness so this flow can be demonstrated even if live FLHSMV is unstable.

The harness should be able to:
- replay sample FLHSMV-like responses
- simulate success responses
- simulate empty responses
- simulate malformed responses
- simulate transient HTTP failures
- simulate duplicate incidents
- demonstrate DB insertion + lead creation

This test harness must prove:
- parser works
- inserts work
- dedupe works
- lead creation works
- logs are emitted

If live data works, great. If live data is unstable, the harness still must provide hard proof.

- leads are actually created from qualifying crashes
