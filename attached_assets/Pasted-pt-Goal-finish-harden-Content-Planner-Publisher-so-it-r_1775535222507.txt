## Scope — exactly what to implement and harden

### A — Scheduler & Worker (server)
1. **Startup integration**
   - Wire `processDueScheduledPosts()` into server startup (`server/index.ts` or equivalent) so a background loop runs when `DISABLE_BACKGROUND_WORKERS` is `false`.
   - Use a resilient background pattern: `setInterval` (configurable interval) **plus** distributed lock (DB advisory lock or Redis lock) so multiple app instances don’t run the same jobs concurrently.

2. **Job model & lifecycle**
   - Use existing `content_publishing_jobs` table as the authoritative job queue. If missing fields, add: `id, content_post_id, platform, scheduled_at_utc, attempt_count, status (queued|processing|published|failed|skipped|cancelled), last_error, started_at, finished_at, lock_owner, lock_expires_at, created_at, updated_at, external_post_id`.
   - Job processing semantics:
     - Claim job atomically (UPDATE ... WHERE status='queued' AND scheduled_at <= now AND lock_owner is null -> set status='processing', lock_owner=instanceId, lock_expires_at=now+heartbeat).
     - Heartbeat: worker refreshes `lock_expires_at` while processing; if heartbeat lost, other workers can recover job after expiry.
     - On success: set `status=published`, write `external_post_id`, `finished_at`.
     - On failure: increment `attempt_count`, set `last_error`, set status to `queued` with exponential backoff or `failed` after max attempts (configurable).
     - Provide idempotency: dedupe by `external_post_id` and by a unique job fingerprint.

3. **Publishing flow**
   - For each platform (FB/IG, TikTok if integration exists):
     - Validate integration and OAuth token before attempting.
     - Upload media using platform-specific flows (image/video). Retry transient upload failures.
     - Create the post with scheduled publish when API supports it, or publish immediately at scheduled moment.
     - Respect platform rate limits: exponential backoff and retry with jitter.
     - If token expired, attempt refresh and retry once; if refresh fails, mark job failed and notify operator.
   - Fallback behavior: if publish to primary platform fails permanently but others succeed, mark job partially succeeded and surface failure in UI.

4. **Readiness checks before scheduling**
   - When creating `content_publishing_jobs`, validate that the account has valid platform credentials. If not, job should be created with `status=failed_setup` and UI should show actionable CTA (Connect account).
   - Store `timezone` and schedule in UTC internally; allow user timezone input on create.

5. **Metrics & Observability**
   - Emit metrics / DB counters: jobs_processed_total, jobs_success_total, jobs_fail_total, publish_latency_seconds, queue_length.
   - Record full request/response traces for external calls into `integration_events` or `webhook_delivery_logs` (redact secrets).

6. **Testing & safety**
   - Ensure safe dry-run mode (TEST_PUBLISH=true) that simulates external API responses for E2E testing.
   - Provide `POST /api/debug/publish-test` to trigger a publish for a given post for manual verification.

   - `POST /api/content/publishing-jobs/:jobId/cancel` — cancel
   - Add health endpoint `GET /api/health/content-publisher` that returns scheduler status and queue stats.

### D — Acceptance Criteria (Must pass before you mark done)
**Backend**
- Scheduler loop runs automatically in production (when `DISABLE_BACKGROUND_WORKERS=false`) and claims jobs with distributed lock across instances.
- `content_publishing_jobs` records are created when scheduling a post.
- Worker processes jobs, updates job `status`, `attempt_count`, and `external_post_id` (or `last_error` on fail).
- Idempotency: re-running a job for the same scheduled timestamp does not produce duplicate posts on external platforms.
- Retry/backoff behavior implemented; max attempts configurable (default 5).
- Robust logging: each external call logged to `integration_events` or `webhook_delivery_logs`.
- Metrics and health endpoint implemented.

**Frontend**
- Content Planner page is reachable from nav, mobile-friendly, and matches Apex visual standard.
- Composer allows scheduling and previewing per platform, creates jobs, and shows status in UI.
- Admin can view and manage publishing jobs and retry/cancel failed ones.
- Empty/loading/error states implemented and polished.

**Testing / Proof**
- Create one test post (with small image) scheduled for immediate publish; verify:
  - `content_publishing_jobs` row created
  - Worker processed job within < 60s
  - API calls to external platform logged
  - `status=published` and `external_post_id` saved
  - UI shows the post as `Published` and links to external post (if available)
- Create a test post queued for future time (5 minutes in future); verify job waits and then processes at scheduled time.
- Simulate platform transient failure and verify retry/backoff then eventual success or `failed` after max attempts.
- Simulate OAuth token expiry, verify token refresh attempted and failure path not silent.
- Run concurrency test: Simulate two worker instances; ensure job is processed exactly once (claim + heartbeat logic prevents double runs).
- Timezone test: schedule in alternate user timezone; ensure final publish time aligns.

**Edge cases to cover**
- Media upload failures (oversize, invalid format)
- OAuth revoked or expired
- Platform rate-limit error (429)
- Duplicate schedules for same content
- Network failures mid-upload
- Multiple platforms (one fails, others succeed)
- Server restarts while processing a job (job recovers after lock expiry)

### E — Implementation notes & constraints
- Respect `DISABLE_BACKGROUND_WORKERS` flag. When true, scheduler must not start.
- Use existing `content_publishing_jobs` and `webhook_delivery_logs` tables if present. If you add columns/tables, document schema migrations.
- For distributed locking, prefer DB advisory locks if Redis not available. Provide fallback.
- Use UTC in DB; convert to user timezone only for UI display.
- Redact secrets in logs; keep request/response bodies but mask tokens.
- Keep code modular: `publisher/scheduler.ts`, `publisher/worker.ts`, `publisher/platforms/meta.ts`, `publisher/platforms/tiktok.ts`, `api/content/*`.
- Keep tests in `tests/content_publisher/*.spec.ts` and include scripts to run them.
